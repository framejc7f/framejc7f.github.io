# Финансовый помощник

Статическое фронтенд-приложение для личного учёта финансов: доходы, расходы, обязательные платежи, возможный доход, календарь и базовая аналитика.

Проект рассчитан на запуск как обычный статический сайт, в том числе на GitHub Pages. Данные пользователей хранятся в Supabase, а доступ к приложению закрыт авторизацией.

## Что есть сейчас

- Отдельная страница входа: `auth.html`
- Закрытая основная страница приложения: `index.html`
- Авторизация пользователей через Supabase Auth
- Синхронизация данных пользователя с Supabase
- Учёт доходов и расходов
- История операций с фильтрами
- Обязательные платежи
- Регулярные платежи: ежедневные, еженедельные, ежемесячные
- Возможный доход с подтверждением
- Календарь операций и платежей
- Базовая аналитика по расходам и динамике
- Экспорт и импорт данных в JSON
- PWA-режим и service worker для production
- Адаптация под мобильные устройства

## Как устроен вход

1. Пользователь открывает сайт.
2. Если сессии Supabase нет, происходит переход на `auth.html`.
3. После входа или регистрации пользователь попадает в `index.html`.
4. Без активной сессии основной функционал недоступен.

## Где хранятся данные

Основное хранилище: Supabase.

В приложении используются:

- Supabase Auth для входа и выхода
- Таблицы Supabase для пользовательских данных
- Локальный кэш в браузере как вспомогательный слой

Локальное хранилище не считается главным источником данных. Оно используется для кэша и промежуточного состояния.

## Точки входа

- `auth.html` — страница авторизации
- `index.html` — основное приложение


## Локальный запуск

Важно: проект использует ES modules, поэтому не открывайте `index.html` двойным кликом как `file://...`.

Запустите локальный сервер из папки проекта:

```powershell
py -m http.server 8000
```

Если `py` не работает:

```powershell
python -m http.server 8000
```

После этого откройте:

- [http://localhost:8000/auth.html](http://localhost:8000/auth.html)

Или просто корень сайта, если сервер отдаёт `auth.html` как стартовую страницу через текущую навигацию проекта.

## Настройка Supabase

Для работы проекта должны быть настроены:

- проект Supabase
- включённый Email provider
- корректные Redirect URL
- таблицы для данных приложения
- Row Level Security
- политики доступа по `auth.uid()`

В проекте клиент Supabase создаётся в:

- `src/app/client.js`

Там используются:

- `SUPABASE_URL`
- `SUPABASE_KEY`

Публичный `publishable` key допустимо хранить во фронтенде, но безопасность данных должна обеспечиваться через RLS и policies.

## Деплой на GitHub Pages

1. Загрузить проект в GitHub-репозиторий
2. Убедиться, что в репозиторий попали:
   - `index.html`
   - `auth.html`
   - `src`
   - `public`
   - `manifest.json`
   - `sw.js`
3. В репозитории открыть `Settings -> Pages`
4. Выбрать ветку публикации
5. Сохранить настройки

После публикации проверьте:

- открывается страница авторизации
- вход работает
- после входа открывается приложение
- синхронизация с Supabase проходит без ошибок

## Импорт и экспорт

Приложение поддерживает:

- экспорт текущих данных в JSON
- импорт JSON-резервной копии обратно в приложение

Это полезно как дополнительный ручной backup, даже при использовании Supabase.
